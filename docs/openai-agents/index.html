OpenAI Agents SDK tracing
that fails the PR.

DProvenanceKit ships a drop-in tracing processor for the OpenAI Agents SDK: register it once and every agent run is recorded as a queryable, diffable trace. Then diff a candidate run against a golden baseline and gate CI when the reasoning drifts — a dropped tool call, a looping step, a changed execution path. Agent regression testing, two lines of setup.

# 01 · install

Install

shell
$ pip install "dprovenancekit[openai-agents]"

Python 3.9+ · zero third-party deps in the core · the extra pulls in the openai-agents package.

# 02 · register

Register the processor

One register() call adds a global tracing processor (via the SDK's add_trace_processor). Everything your agents do afterwards is recorded — no other code changes.

trace_agents.pypython
from dprovenancekit import SQLiteTraceStore
from dprovenancekit.integrations.openai_agents import register, OpenAIAgentsTraceEvent

store = SQLiteTraceStore(OpenAIAgentsTraceEvent, "traces.sqlite")
register(store)   # registers a global tracing processor

# ... run your agents normally; each run is recorded ...

register() returns the processor so you can flush or remove it later. Options: capture_payloads (default on — include tool/generation IO previews in event attributes; turn off to keep only structural metadata like names, models, token counts) and link_lifecycle (default on — emit the provenance edges below). Full reference: the OpenAI Agents SDK section of the README.

# 03 · what gets recorded

What gets recorded

DProvenanceTracingProcessor implements the SDK's TracingProcessor interface, so it receives the same trace/span stream as the SDK's built-in exporter:

  • Each agent run becomes a trace-run (context_id = the trace name).
  • Every span start/end becomes a typed event — agent.start, generation.end, function.start, guardrail.error, … — in execution order.
  • The span's span_id/parent_id become the span tree.
  • The active agent, tool, or model becomes the event's engine.
  • Errors and triggered guardrails are recorded at CRITICAL priority.
  • Lifecycle provenance edges are emitted: DERIVED_FROM (start → end) and INFORMED (parent → child).

One registered processor captures every run; the same fingerprint / diff / align tooling then applies.

# 04 · gate in ci

Then gate it in CI

Once runs are recorded, diff any run against a golden baseline and fail the PR when the reasoning drifts. The server-less gate CLI exits non-zero on regression, and the DProvenanceKit Regression Gate GitHub Action drops it into any workflow — see the full gate story on the main page.
